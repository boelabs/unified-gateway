## Application logs

Structured JSON, one object per line, on stdout/stderr (`src/logging/log.ts`). Ship them to whatever
log collector your platform already uses — no special integration needed. Every request carries
`x-request-id` (accepts an inbound one, generates one otherwise) — **always include it** when
investigating an issue; it's the join key between the application log line, the request log row, and
any OTEL span.

## Request logs (`request_logs`)

Every request writes one row, queryable via `GET /admin/logs` (paginated) or aggregated via
`GET /admin/usage`. Relevant columns:

| Column | What it captures |
|---|---|
| `requestId`, `virtualKeyId`, `publicModel`, `deploymentId`, `adapterKey`, `callType` | Identity of the request and what served it. |
| `status`, `httpStatus`, `error` | Outcome. `error` holds the gateway's classified error (class, code, provider detail) on failure. |
| `promptTokens`, `completionTokens`, `totalTokens`, `costCents` | Usage and its computed cost — see [Cost accounting](/docs/cost-accounting). |
| `durationMs`, `ttftMs`, `upstreamTtftMs` | End-to-end duration, time-to-first-token, and the **upstream-only** TTFT (isolated from the gateway's own overhead) — the same timing [Routing](/docs/routing)'s latency-based strategy uses. |
| `cacheHit`, `retries`, `fallbackUsed` | Whether this was a cache hit (logged with `cost: null` — see [Cost accounting](/docs/cost-accounting)), how many router retries it took, and whether a fallback chain was used. |
| `metadata` | Free-form: deployment `label`, `parameterPolicy`, reasoning `requested`/`effective`/`clamped`, and `contentInputs` native/parsed/materialized counts. |
| `attempts` | The same per-attempt log exposed via `x-unified-routing-metadata` (see [Headers](/docs/headers)) — every deployment tried, not just the winner. |
| `requestBody`, `responseBody` | Truncated at `MAX_STRING_LENGTH_PROMPT_IN_DB` (default 8000 chars) — storage only, never affects what's sent upstream. |

Query with the same filter set on both `/admin/logs` and `/admin/usage`: `virtualKeyId`, `publicModel`,
`deploymentId`, `adapterKey`, `callType`, `status`, `requestId`, `cacheHit`, `start`, `end`. Full route
list: [Admin API reference](/docs/reference-admin-api#logs-and-usage).

## OpenTelemetry

Off by default. Turn it on with:

```bash
OTEL_ENABLED=true
OTEL_EXPORTER_OTLP_ENDPOINT=http://otel-collector:4318
```

- **Metrics** — exported every `OTEL_METRIC_EXPORT_INTERVAL_MS` (default 60s).
- **Traces** — one per request, through the same pipeline described in [Architecture](/docs/architecture).
- **Span events** — `OTEL_LOG_PAYLOADS` (default `true`) attaches the **full, untruncated**
  request/response/error body to span events, independent of the Postgres truncation above. See
  [Security](/docs/security#observability-payloads) before enabling this in a compliance-sensitive
  environment — your OTEL backend now holds data your database deliberately doesn't.
- **Service identity** — `OTEL_SERVICE_NAME` (default `unifiedgateway`), plus the standard OTEL SDK
  variables (`OTEL_EXPORTER_OTLP_HEADERS`, etc.) for anything beyond the endpoint. See
  [Environment variables](/docs/reference-environment#observability-opentelemetry).
